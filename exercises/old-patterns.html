This document shows working examples of the “old” ways of doing common data manipulation tasks that dplyr 1.2.0’s new functions are designed to improve.

Filtering Out Rows (Old Way)

Problem: Using ! negation with filter() to drop rows

Recoding Values with case_when() (Old Way)

Problem: Using case_when() with multiple == checks for simple value mapping

penguins |>
  slice(1:4) |>
  select(species, island, body_mass_g) |>
  mutate(
    island_location = case_when(
      island == "Biscoe" ~ "Southwest",
      island == "Dream" ~ "Northwest",
      island == "Torgersen" ~ "Northwest",
      .default = island
    )
  )

Using case_match() (Old Way - Now Soft Deprecated)

Problem: case_match() is being deprecated in favor of recode_values()

penguins |>

  select(species, island, body_mass_g) |>

Replacing Values with case_when() (Old Way)

Problem: Using case_when() with .default for simple replacements

# Cap body_mass_g at 5000g
penguins |>
  slice(1:4) |>
  select(species, island, body_mass_g) |>
  mutate(
    body_mass_g_capped = case_when(
      body_mass_g > 5000 ~ 5000,
      .default = body_mass_g  # Must specify .default even though it's the same column
    )
  )
# A tibble: 4 × 4

Summary of Pain Points

  1. filter() with !: Complex NA handling, unclear intent
  2. Complex | and &: Hard to read nested logic
  3. case_when() for exact matches: Repetitive == checks
  4. case_when() for replacements: Unnecessary .default boilerplate
  5. case_match(): Being deprecated, used for simple value mapping

All of these patterns work, but dplyr 1.2.0 provides clearer, more expressive alternatives!
